# Page Purpose Boundaries

## Purpose
Prevent pages from drifting into “everything pages.”

Strategic rule: **each page primarily showcases what that page is for.**

Related entities (services, proofs, tools, problems) may support the page, but must not dominate it.

---

## Page-primary-object rule

Before writing or revising any page, explicitly name:

- **Page type**
- **Primary object**
- **Primary question the page answers**
- **Primary CTA**

If these are not clear, the page plan is incomplete.

---

## Boundaries by page type

### Service detail pages
- **Primary object:** the service (what changes, who it’s for, how it works, what outcomes look like)
- **Primary question:** “Is this the right implementation lane for my situation?”
- **Primary CTA:** evaluate-stage action (roadmap / fit conversation / start here)
- **Must not drift into:** a relationship map of every problem/proof/tool connected to the service; long encyclopedic capability lists

### Proof / project detail pages
- **Primary object:** the proof story (pre-state → intervention → outcomes → why it worked)
- **Primary question:** “What changed, and how did it change the system?”
- **Primary CTA:** “See the relevant implementation path” or “Diagnose your state”
- **Must not drift into:** client promotion, brand celebration, or a list of unrelated services/tools “used”

### Problem detail pages
- **Primary object:** the problem (symptoms, stakes, system breakdown, the fix shape)
- **Primary question:** “Is this what’s happening to me, and what should I do next?”
- **Primary CTA:** diagnose-stage action (tool/assessment)
- **Must not drift into:** a service page in disguise; long blog-style explanation; surfacing many services

### Tool pages
- **Primary object:** the diagnostic tool (what it measures, how to run it, how to interpret results)
- **Primary question:** “What does this tool diagnose, and where does it route me?”
- **Primary CTA:** run/complete the tool
- **Must not drift into:** long educational essays, service overviews, or proof galleries

### Hub pages (Problems / Proof / Tools / Services)
- **Primary object:** curated navigation and selection
- **Primary question:** “Where should I click next?”
- **Primary CTA:** a single next step for the hub’s job (diagnose, proof, implementation)
- **Must not drift into:** trying to summarize every detail of every child page

---

## Reduction rule (default)

When a page has many valid connected entities:

1. pick the **few** that best reinforce the primary object
2. show them **after** the main narrative is complete
3. keep the UI small (supporting, not dominant)
4. prefer a “**See more**” jump to the relevant hub instead of expanding the page
