# Page Generation Rules

## Purpose
Define how the agent system should plan, generate, review, and score future pages.

This file exists to stop generic page generation and force stronger strategic alignment before implementation.

## Non-Negotiable Rules
1. Do not generate generic agency pages.
2. Do not write long service-menu pages without a clear problem and buyer-state frame.
3. Do not imply that company size alone determines fit.
4. Do not ignore the missing-system buyer path.
5. Do not generate copy that sounds bigger, softer, or less accountable than the actual business.

## Every Page Must Clarify These Things
- what problem state is being addressed
- which buyer state is being served
- why the issue matters commercially
- what the next step is
- what proof supports the promise

## Required Inputs Before Generating a Page
The agent should identify:
- page type
- primary buyer state
- trust stage of the page
- primary cluster
- supporting proof path
- recommended CTA pattern

If those are not clear, the page plan is incomplete.

## Page-Type Rules

### Homepage
Must:
- support both broken-system and missing-system buyers
- provide a clear self-selection path
- show proof and system logic early
- offer both low-friction and high-intent CTAs

Should not:
- read like a generalist agency hero
- bury the practical path for smaller valid buyers
- rely on abstract positioning without scannable paths

### Problem Pages
Must:
- describe the problem in business terms
- show symptoms, stakes, and fix logic
- connect the problem to proof, tools, and services
- make the next step obvious

Should not:
- become long generic service sales pages
- list unrelated services
- explain the problem without showing what changes

### Service Pages
Must:
- explain what system change is made
- identify which buyer state the page is for
- connect to proof and process
- clarify outcomes in practical terms

Should not:
- read like a feature brochure
- sound like a generic agency capability page
- rely on buzzwords instead of concrete business logic

### Proof Pages
Must:
- make the result scannable quickly
- explain the original problem state
- show what was built or fixed
- bridge to related problems, services, or tools

Should not:
- celebrate metrics without context
- hide the operational or system-level change
- become disconnected from the rest of the site logic

### Resource Pages
Must:
- teach something useful
- support trust progression
- bridge to problems, tools, or proof

Should not:
- exist as isolated content islands
- talk in generic trend language without relevance to actual buyers

## Buyer-State Rules
### Broken-System Pages
Emphasize:
- diagnosis
- integration depth
- repair and optimization
- strategic ownership

### Missing-System Pages
Emphasize:
- practical foundation
- simplification
- usable setup
- reduced owner burden
- connected first version of the system

## Proof Rules
Every strategic page should have a proof plan, even if proof is not fully shown on the page.

Questions to answer:
- what proof route reinforces this page?
- what type of outcome supports it?
- what metric or practical result makes the promise credible?

## CTA Rules
Every page should choose one primary CTA based on trust stage and buyer state.

Optional secondary CTAs are allowed only if they support progression cleanly.

## Failure Patterns to Block
- page sounds like a generic full-service agency
- page ignores one of the valid buyer-state paths when it should not
- copy implies size-based disqualification
- no commercial stakes are explained
- no proof path is connected
- CTA is too high-trust for the page context
- service explanation is channel-first instead of system-first

## Review Questions
Before approving a page draft, ask:
- does this sound like Darling MarTech specifically?
- does it explain a system problem or just mention tactics?
- would a small but valid operator still feel seen if appropriate?
- would a complex buyer still trust the depth?
- is the next step correct for the reader’s likely trust stage?
