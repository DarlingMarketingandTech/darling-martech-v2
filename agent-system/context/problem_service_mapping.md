# Problem -> Service Mapping

Use this matrix to keep recommendations coherent across problems, services, proof, tools, and CTA types.

## Problem State: Fragmented stack, unclear attribution, execution drag
- Buyer state: **broken-system**
- Service clusters: `Scale`, `Build`
- Likely tools: growth bottleneck quiz, attribution snapshot, stack grader
- Likely proof path: implementation/system restructuring case studies
- CTA type: diagnostic call or technical roadmap

## Problem State: No coherent system, manual operations, low clarity
- Buyer state: **missing-system**
- Service clusters: `Foundation`, `Build`
- Likely tools: growth bottleneck quiz, system-foundation diagnostic
- Likely proof path: practical setup and operational uplift case studies
- CTA type: scoped foundation consult / roadmap

## Problem State: Weak conversion journey and low trust movement
- Buyer state: both
- Service clusters: `Experience`, `Brand`
- Likely tools: funnel/UX diagnostic, conversion audit
- Likely proof path: conversion-focused proof and decision-flow improvements
- CTA type: evaluate-stage consultation or targeted audit

## Problem State: Low visibility and unreliable qualified demand
- Buyer state: both
- Service clusters: `Grow`, `Brand`
- Likely tools: visibility/readiness diagnostics, demand audit
- Likely proof path: pipeline growth and qualified demand proof
- CTA type: diagnostic + implementation path CTA

## Mapping Rules
- Always map page recommendations to one explicit buyer state (or both with separated paths).
- Include at least one tool and one proof linkage for each recommendation.
- Do not recommend service structures that leave CTA path undefined.
