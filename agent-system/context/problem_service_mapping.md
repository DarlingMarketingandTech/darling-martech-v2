# Problem / Service / Proof / Tool Mapping

## Purpose
Provide a coherent mapping layer so the agent system can recommend pages, services, proof, and CTAs without breaking the site’s internal logic.

This mapping should be used when planning:
- problem pages
- service pages
- homepage wayfinding
- CTA logic
- proof cross-links
- tool recommendations

## Buyer State Layer
Before choosing services or tools, identify the buyer state:

### Buyer State A — Broken System
The business has a stack, process, or team, but it is fragmented, underperforming, or untrustworthy.

### Buyer State B — Missing System
The business does not have a real operating system yet.
It may have a website and tools, but not a connected capture → follow-up → conversion → visibility structure.

## Problem Mapping

### 1. No one is steering the whole system
Primary buyer state:
- broken system

Primary clusters:
- Scale
- Build

Likely services:
- fractional CMO / strategic ownership
- roadmap / audit work
- positioning + system prioritization

Likely proof:
- strategic rebuilds
- roadmap-driven improvements
- cases where disconnected execution was unified

Likely tools:
- growth bottleneck quiz
- build vs buy stack calculator
- roadmap generators

CTA direction:
- diagnose the real bottleneck
- review process / roadmap path
- book a strategic conversation

## Mapping Rules for the Agent
- Never recommend a service page that is disconnected from a recognizable problem state.
- Never recommend proof without explaining why it is relevant to the problem and buyer state.
- Never recommend a CTA without matching it to trust stage and buyer state.
- If the buyer state is missing system, prefer practical/productized framing before enterprise framing.
- If the buyer state is broken system, emphasize diagnosis, integration depth, and strategic repair.
