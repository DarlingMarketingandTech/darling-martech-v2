# Service Clusters

## Purpose
Define the recommended cluster architecture the site and agent system should use when organizing services, pages, proof, and tools.

The goal is to prevent the site from becoming a long list of unrelated services and to support both major buyer states:
- broken / fragmented systems
- missing / underbuilt systems

## Cluster Model Overview
The clusters should communicate business outcomes and operating logic, not departments.

Recommended primary cluster set:
1. **Build** — brand, site, conversion, digital experience, structural trust
2. **Scale** — CRM, automation, attribution, operations, reporting, pipeline systems
3. **Grow** — visibility, demand capture, search, channel performance, content systems
4. **Foundation** — first real system for smaller or system-starved operators

## Cluster 1 — Build
### Purpose
Improve how the business is represented, understood, and converted through its website, messaging, and experience layer.

### Typical buyer state
- brand quality does not match real service quality
- site gets traffic but underperforms
- first impression weakens trust
- structure is confusing or generic

### Strong outcomes
- stronger conversion
- clearer positioning
- better trust signals
- better first-screen clarity
- site architecture aligned to buyer paths

### Likely services
- positioning and messaging
- website strategy and rebuild
- conversion UX
- brand identity / system refinement
- page architecture / CTA structure

### Likely proof connections
- project pages where site conversion, experience clarity, or trust improved
- before/after proof showing stronger structure or booking results

### Likely tool connections
- funnel ROI tools
- homepage / page audit tools
- conversion diagnostics

## Cluster 2 — Scale
### Purpose
Build or repair the infrastructure that captures, routes, follows up on, and measures demand.

### Typical buyer state
- CRM distrust
- disconnected tools
- manual follow-up
- attribution blind spots
- operational drag across lead-to-close flow

### Strong outcomes
- cleaner pipeline
- less manual work
- stronger follow-up
- attribution clarity
- better operational leverage

### Likely services
- CRM architecture
- workflow automation
- martech stack build
- attribution and analytics
- pipeline automation
- reporting / dashboard design

### Likely proof connections
- automation builds
- martech consolidation work
- attribution layers
- system integrations

### Likely tool connections
- martech fragmentation scorecard
- automation savings calculator
- attribution clarity analyzer
- stack calculators

## Cluster 3 — Grow
### Purpose
Increase qualified demand and visibility through search, content, local presence, and channel economics.

### Typical buyer state
- buyers exist but do not find the business
- branded demand is weak
- search visibility lags business quality
- paid or organic channels are underperforming

### Strong outcomes
- stronger visibility
- more qualified traffic
- better demand capture
- improved CAC / channel economics
- more consistent local or category presence

### Likely services
- local SEO
- GEO / AI-search readiness
- content systems
- paid media management
- demand capture strategy

### Likely proof connections
- local visibility improvements
- traffic or booking growth
- demand capture improvements

### Likely tool connections
- geo readiness auditor
- demand capture / CAC tools
- growth bottleneck quiz

## Cluster 4 — Foundation
### Purpose
Provide a practical first real operating system for smaller firms and other system-starved operators.

### Typical buyer state
- brochure-style site
- weak or absent CRM
- disconnected booking/forms/reviews
- manual follow-up
- owner-dependent processes
- expensive or underused tools with no operating logic

### Strong outcomes
- fewer missed leads
- stronger intake and booking flow
- practical automation
- cleaner visibility into what is working
- simpler, more usable stack

### Likely services
- system foundation build
- launch stack setup
- intake / booking setup
- CRM setup and cleanup
- review / follow-up automation
- practical conversion-focused site or landing structure

### Likely proof connections
- local service projects
- smaller-firm workflow / intake / booking improvements
- system simplification proof

### Likely tool connections
- missed lead calculators
- intake health checks
- stack simplification audits
- booking / review flow audits

## Mapping Rules
- Each problem should map to one primary cluster and one secondary cluster at most.
- Each service page should belong to one primary cluster.
- Each proof item should reinforce at least one cluster.
- Each tool should help users self-diagnose a cluster-relevant issue.

## IA Recommendation
The site does not need to expose all cluster language directly in the primary nav right away.

But the agent system should use these clusters to:
- plan service pages
- generate internal cross-links
- keep proof mappings coherent
- avoid generic service sprawl
- support future split-path navigation for broken-system vs missing-system buyers
