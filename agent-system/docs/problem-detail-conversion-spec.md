# Problem Detail Conversion Spec

## Purpose
Define how problem detail pages should be structured for maximum clarity, trust, and conversion.

These pages are decision points, not explanation pages.

## Page Job
Each problem page must:
- help the visitor recognize themselves quickly
- explain the cost of the problem
- clarify the system-level failure
- show how the system is fixed
- route the visitor to the correct next step

## Required Structure

### 1. Hero
- clear problem identification
- short supporting explanation
- primary CTA (diagnose)
- secondary CTA (proof or service)

### 2. Symptoms
- short, scannable bullets
- written so users immediately recognize their situation

### 3. What This Actually Costs
Explain real impact:
- lost leads
- wasted spend
- missed opportunities
- time and operational drag

Keep concise but concrete.

### 4. System Breakdown
Explain:
- what is actually broken
- why surface fixes fail
- what layer of the system is failing

Avoid generic language.

### 5. System Fix
Explain:
- what changes in the system
- how components become connected
- what outcome behavior looks like

### 6. Next Best Step Module
Must include:
- diagnose (tool)
- learn (proof)
- evaluate (service direction)

This is the core routing mechanism.

### 7. Proof Bridge
- show relevant outcome examples
- make proof feel directly applicable

## UX Rules
- high scanability
- strong section separation
- clear hierarchy
- minimal filler content
- strong CTA clarity

## Failure Patterns
Block when:
- page reads like a blog article
- no cost/stakes section
- weak or missing routing
- CTA is unclear
- system logic is not explained
